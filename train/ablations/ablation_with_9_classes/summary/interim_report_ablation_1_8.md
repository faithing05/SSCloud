# Промежуточный аналитический отчет по серии ablation-экспериментов (1-8)

## 1. Статус и цель документа

Настоящий документ фиксирует промежуточные результаты экспериментальной серии ablation (итерации 1-8) по задаче многоклассовой сегментации. Отчет подготовлен в формально-деловом стиле для последующего использования в научных публикациях, внутренней отчетности и планирования следующего этапа исследований.

Основные цели текущего этапа:

1. Проверка устойчивости базовой архитектуры U-Net-подобного семейства и ее модификаций.
2. Анализ влияния методов компенсации дисбаланса классов (class weights, oversampling и гибридные схемы).
3. Оценка влияния масштаба входного изображения и дополнительных блоков (transformer/attention) на качество и ресурсоемкость.
4. Фиксация ограничений по видеопамяти (OOM) и подготовка инфраструктурных мер.

Источник данных: логи и сводки `train/summary_run_ablation_1.md` ... `train/summary_run_ablation_7.md`, `train/summary_report_ablation_5.md`, сценарий `train/run_ablation_8.sh`.

## 2. Контур проведенных экспериментов

### 2.1. Общие условия

- Основной режим: многоклассовая сегментация, в большинстве прогонов `batch_size=1`, `scale=0.1`, `num_workers=4`.
- Частые длительности прогонов: 10, 20 и 50 эпох.
- Ключевые метрики: Validation Dice, Mean IoU (без NaN), Per-class IoU, среднее время инференса на изображение.
- Валидационная выборка мала (`Validation size = 13`), что повышает дисперсию метрик между соседними прогонами.

### 2.2. Варианты ablation

В серии последовательно тестировались:

1. Базовая конфигурация (baseline).
2. Class weights (жесткие и смягченные).
3. Oversampling (обычный, облегченный и редкоклассовый сбалансированный).
4. Гибридные комбинации weights + sampler.
5. В ablation 8 - рост масштаба входа (`scale 0.2/0.25`) и гипотеза усиления редких классов на более детализированных изображениях в конфигурации с transformer/attention.

### 3.2. Стабильные тенденции по всей серии

1. Baseline обеспечивает наиболее устойчиво высокий Dice на коротких и длинных прогонах.
2. Политики class weights в жестком виде чаще ухудшают глобальные метрики.
3. Смягченные веса и мягкий sampler в ряде запусков поднимают Mean IoU, но обычно с компромиссом по Dice.
4. Комбинированные схемы weights + sampler преимущественно показывают деградацию по Dice.

## 4. Интерпретация дисбаланса классов

### 4.1. Эмпирические признаки дисбаланса

Во множестве прогонов наблюдается регулярная картина:

- Для части классов IoU стабильно близок к нулю (например, классы 5, 7, 8 в большинстве отчетов).
- Отдельные классы имеют высокую волатильность между конфигурациями.
- Улучшение агрегированных метрик на частых классах не гарантирует прогресс по редким.

Данный паттерн согласуется с гипотезой выраженного дисбаланса по пиксельной доле классов и недостаточной представленности редких классов в обучении.

### 4.2. Практическая трактовка

1. Рост Mean IoU в отдельных экспериментах не является достаточным основанием для выбора модели без контроля Dice и per-class стабильности.
2. Для production-ориентированного режима приоритет сохраняется за baseline-подходом до внедрения более точной редкоклассовой стратегии.
3. Для научной части необходимо отдельно документировать trade-off "global Dice vs rare-class fairness".

## 5. Отдельная фиксация по попытке 8 (ресурсные ограничения)

### 5.1. Гипотеза и дизайн ablation 8

Сценарий `train/run_ablation_8.sh` предусматривает:

- переход к конфигурации с `--use-transformer --use-attention`,
- проверку масштабов `scale=0.1`, `0.2`, `0.25`,
- дополнительный вариант с мягкими class weights на `scale=0.2`.

Заявленная исследовательская гипотеза: увеличение масштаба входа может улучшить сегментацию малых/редких объектов за счет сохранения пространственных деталей.

### 5.2. Зафиксированное ограничение

По факту выполнения серии на текущей машине зафиксирована ошибка нехватки видеопамяти (OOM) на 8-й попытке. Это прервало полноценную валидацию гипотезы увеличенного масштаба и не позволило завершить сопоставимое сравнение по всем запланированным конфигурациям.

### 5.3. Управленческое решение

Принято решение перенести дальнейшее обучение на машину с большей доступной VRAM. Это решение является корректным и необходимым для:

1. воспроизводимого запуска scale-sensitive экспериментов,
2. тестирования альтернативных архитектур без искусственного урезания модели,
3. снижения рисков преждевременного завершения прогонов.

## 6. Промежуточные выводы по архитектуре

1. Текущий контур близок к U-Net-подобной парадигме и демонстрирует приемлемую стабильность на baseline-конфигурации.
2. Простое усиление дисбаланс-коррекции (weights/sampler) без архитектурных изменений не дало устойчивого превосходства.
3. Следующий этап обоснованно требует архитектурного расширения (UNet++ / DeepLabV3+ / SegFormer / FPN как референс), особенно в контексте редких классов и мелких структур.

## 7. План следующего этапа (утвержденный)

Ниже зафиксирован план, предложенный по итогам обсуждения.

### 7.1. Блок "стабилизация"

1. Сформировать новый baseline на более мощной машине (без OOM), сохранив прежние split/seed для сопоставимости.
2. В обязательные KPI включить: Dice, Mean IoU, Macro IoU, IoU редких классов.
3. Зафиксировать baseline как контрольную точку для всех последующих запусков.

### 7.2. Трек 1 - дисбаланс классов (без смены архитектуры)

1. Тест loss-функций: `CE + Dice` с мягкими классовыми весами.
2. Далее тест `Focal` или `Tversky` для усиления редких классов.
3. Внедрение class-aware sampling (oversampling редкоклассовых примеров).
4. Принцип проведения: изменять по одной оси за запуск (loss или sampler), чтобы корректно оценивать вклад.

### 7.3. Трек 2 - альтернативные архитектуры

1. `UNet++`.
2. `DeepLabV3+`.
3. `SegFormer-B0/B1`.
4. `FPN + ResNet34/50` как стабильный инженерный референс.

Условие честного сравнения: использовать лучший найденный setup из Трека 1 (loss/sampler) без дополнительных изменений.

### 7.4. Меры против OOM

1. Сохранить mixed precision (`--amp`).
2. Использовать gradient accumulation для роста эффективного batch без роста VRAM.
3. При необходимости перейти на patch/tile обучение.
4. Снижать базовую ширину каналов в архитектурах-кандидатах.
5. Включать gradient checkpointing в тяжелых блоках.

### 7.5. Приоритетная последовательность запусков (6-8 экспериментов)

1. Baseline на новой машине (контроль).
2. Baseline + weighted `CE+Dice`.
3. Baseline + class-aware sampler.
4. Лучшая комбинация из п.2 и п.3.
5. `UNet++` + лучший setup дисбаланса.
6. `DeepLabV3+` + лучший setup дисбаланса.
7. `SegFormer-B0/B1` + лучший setup дисбаланса.
8. Финальный тюнинг лучшего кандидата (`lr/scheduler/augmentations`).

## 8. Рекомендации для оформления статьи

Для публикационной версии рекомендуется сохранить следующую структуру раздела экспериментов:

1. Описание проблемы дисбаланса классов с ссылкой на per-class IoU и доли пикселей.
2. Таблица ablation по методам балансировки (weights/sampler/hybrid).
3. Отдельный раздел о вычислительных ограничениях (OOM) как факторе дизайна эксперимента.
4. Переход к архитектурным сравнениям как логичное продолжение, а не разрозненный поиск.
5. Прозрачная фиксация компромисса "качество редких классов vs глобальная метрика".

## 9. Текущее управленческое решение

1. На текущем этапе baseline остается опорной конфигурацией для воспроизводимости.
2. Дисбаланс классов подтвержден эмпирически и требует целевого решения.
3. Запуски с повышенным scale и альтернативными архитектурами продолжаются на машине с увеличенным объемом VRAM.
4. Настоящий отчет считается официальной промежуточной фиксацией результатов на дату подготовки документа.
